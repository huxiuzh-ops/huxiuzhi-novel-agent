---

## 0. 结论前置

UI 不是单纯"把数据摆出来"。UI 在这个项目里应该承担的是：**把 AI 的工作过程、项目状态、风险点、决策点，变成作者可掌控的工作台。**

所以它不是一个普通展示页，而是一个：
- 创作台
- 项目控制台
- 状态面板
- 决策入口
- 风险提示层

---

## 1. UI 的产品定位

### 1.1 它不是"可有可无的小页面"

如果这个项目要成为真正可长期使用的小说 Agent，UI 的价值非常大。因为长篇创作不是一次性问答，而是：
- 长周期
- 多状态
- 多文件
- 多角色
- 多决策
- 多版本

单靠聊天窗口很快会失控。

### 1.2 UI 的核心目标

| 目标 | 说明 |
|------|------|
| **状态感知** | 让作者快速知道项目现在是什么状态 |
| **工作入口** | 让作者方便进入当前工作点 |
| **过程可见** | 让作者可以看见 AI 在做什么 |
| **决策入口** | 让作者可以在关键点做决策 |
| **状态修正** | 让作者能修正系统状态 |

---

## 2. UI 总体信息架构

第一版建议采用 8 个主页面 / 模块：

1. Dashboard
2. Chapter Studio
3. Beat Board
4. Timeline
5. World Bible
6. Decision Center
7. Reports & Trace
8. Settings

## 3. Dashboard 详细设计

**Dashboard 是首页**，目标是：作者一打开就知道"项目健康吗？现在最该看哪？"

### 3.1 应展示的信息

**A. 项目总览卡**
- 小说标题、当前卷、当前章节
- 总章节数、草稿数 / 定稿数
- 最近更新时间

**B. 当前工作重点卡**
- 当前建议处理的任务
- 最近生成的章节、待审章节、待修章节、待决策事项数

**C. 伏笔状态卡**
- pending 数、due_soon 数、overdue 数、resolved 数
- 最近最危险的 3 条伏笔

**D. 风险卡**
- 一致性 warning 数
- timeline 风险数
- world 冲突数
- 审稿失败数

**E. 角色 / 世界动态卡**
- 最近新增角色、地点、势力
- 最近更新的规则

**F. Agent 活动卡**
- 最近一次 / 三次任务（写第 8 章、审第 8 章、更新 world rule）
- 状态：done / waiting_human / failed

### 3.2 Dashboard 的真正用途

帮助作者快速回答：
- 现在最该处理什么？
- 有没有危险正在发生？
- AI 最近干了什么？
- 项目状态有没有跑偏？

## 4. Chapter Studio 详细设计

**这是 UI 的核心页面**。如果这个页面做好了，产品就有"创作工作台"的味道了。

### 4.1 页面目标

作者围绕"某一章"进行的所有动作，最好都能在这里完成：
- 看计划
- 看草稿
- 看修订
- 看审稿意见
- 看本章时间线
- 看本章伏笔推进
- 看本章新增实体
- 看本章风险

### 4.2 页面布局建议

```
┌─────────────┬──────────────────────────┬──────────────┐
│  章节导航   │         中栏              │  状态侧边栏  │
│  (左栏)     │      (章节主体)          │  (右栏)      │
└─────────────┴──────────────────────────┴──────────────┘
```

**左栏：章节导航**
- 章节列表：ch001、ch002、ch003...
- 带状态标签：planned / draft / reviewing / revised / final
- 带风险小点：黄色 warning、红色 high risk

**中栏：章节主体（Tab 切换）**

| Tab | 内容 |
|-----|------|
| Plan | Planner 输出的 mini plan：目标、开头、中段、转折点、结尾钩子、必须推进的 beats、风险提醒 |
| Draft | 当前草稿全文 |
| Revisions | 修订版本：v1 / v2 / final |
| Diff | 版本对比：哪一段改了、是 Writer 自改还是按 Editor 建议改 |

**右栏：章节状态侧边栏**

| 分区 | 内容 |
|------|------|
| 基本信息 | 章节标题、字数、状态、最后修改时间 |
| 本章摘要 | 1 段摘要 |
| 本章角色 | 出场角色列表 |
| 本章地点 | 场景地点列表 |
| 本章伏笔 | 推进了哪些、新埋了哪些、哪些没推进但本应推进 |
| 风险提示 | 一致性风险、时间线风险、风格问题、world 冲突 |
| 决策点 | 如果本章触发决策，直接显示入口按钮 |

### 4.3 Chapter Studio 背后的底层能力

> 这里很多看起来像 UI 的功能，底层其实都依赖状态层和编排层。

| UI 要展示 | 底层必须有 |
|-----------|-----------|
| 计划 | Planner 的结构化输出 |
| 风险 | Editor 的结构化 issues |
| 本章角色/地点/伏笔 | chapters index + characters index |
| 本章时间线 | timeline index |

**本质：Agent 层能力的可视化集成页。**

## 5. Beat Board 详细设计

**这页是整个产品差异化很强的一页**。因为"伏笔追踪"是长篇写作里很有痛点、又很适合 AI 干的事情。

### 5.1 页面目标

帮助作者看清楚：
- 哪些伏笔已经埋下
- 哪些快到回收节点
- 哪些已经逾期
- 哪些已回收
- 哪些可能要废弃

### 5.2 推荐视图：看板式

| 列 | 状态 |
|----|------|
| Planted | 已埋入 |
| Active | 推进中 |
| Due Soon | 即将到期 |
| Overdue | 已逾期 |
| Resolved | 已回收 |
| Abandoned | 已废弃 |

**每张卡片显示**：beat id、简述、planted chapter、planned payoff、priority、关联角色

### 5.3 点开单个 beat 详情

- 伏笔描述、埋入章节
- 最近一次被提及
- 计划回收章节、当前状态
- 关联角色、关联地点
- 风险说明
- 被哪些章节推进过

### 5.4 可以支持的操作

- 手动改状态
- 改计划回收章节
- 标记为废弃
- 添加备注
- 跳转到相关章节

### 5.5 这页的价值

> "这个 AI 不只是会写，它真的在帮我管长线结构。"

## 6. Timeline 详细设计

**这个页面是"长期稳定性"的关键**。

### 6.1 页面目标

让作者知道故事时间和事件演进有没有乱。

### 6.2 推荐视图

支持三种切换：

**视图 A：按章节**
- ch001 → ch002 → ch003
- 每章下列出关键事件

**视图 B：按角色**
- 该角色在哪些章节出现
- 每次在哪、知道了什么、发生了什么状态变化

**视图 C：按地点**
- 哪些事件发生在这里
- 哪些角色来过
- 最近一次出现在哪章

### 6.3 风险提示

Timeline 页必须支持显示这些风险：
- 同一时间角色出现在两个地方
- 某角色提前知道不该知道的信息
- 受伤状态没有延续
- 已死亡角色又出现
- 某道具丢失后无故又出现

### 6.4 为什么这页重要

它不是"锦上添花"，而是未来高级审稿能力的可视化入口。很多作者不需要天天看 ontology，但会非常需要：
- 时间顺不顺
- 人物有没有瞬移
- 信息线有没有穿帮

## 7. World Bible 详细设计

**这个页面是"世界设定控制台"**。

### 7.1 子模块划分

- Characters
- Locations
- Factions
- Rules

### 7.2 Characters 页

每个角色卡显示：
- 名字、角色类型、状态
- 所属势力、最近出场章节
- 标签、简短摘要

点开后看详细档案：基本信息、背景、关系、关键事件、当前状态、相关章节

### 7.3 Locations 页

显示：地点名、区域、类型、最近出场、简要介绍

### 7.4 Factions 页

显示：势力名、势力摘要、核心成员、敌对 / 联盟关系、首次出场

### 7.5 Rules 页

显示世界规则摘要：力量体系、地理约束、特殊规则、禁止事项、最近变更记录

### 7.6 World Bible 的作用

它一半是给人看，一半是给系统做"可修正入口"。
- 作者可以查
- 也可以修
- 比如 AI 抽错角色状态，作者能改回来

## 8. Decision Center 详细设计

### 8.1 页面目标

集中处理所有需要作者拍板的高影响事项。

### 8.2 为什么必须单独做页面

如果把决策点散落在聊天或章节页里，长期会乱。独立做 Decision Center 有几个好处：
- 所有待决策事项集中
- 可以看优先级
- 可以看影响范围
- 可以追溯历史决策

### 8.3 列表页字段

每个待决策项显示：标题、类型、风险等级、涉及章节、当前状态、推荐方案、最后更新时间

### 8.4 详情页字段

| 分区 | 内容 |
|------|------|
| **问题是什么** | 例如："是否让林月在 ch012 提前得知实验真相？" |
| **为什么重要** | 例如："会削弱 ch018 的主要反转" |
| **当前上下文** | 当前章节、相关角色、相关伏笔、相关 timeline |
| **方案列表** | 方案 A / B / C：优点、缺点、影响范围 |
| **推荐方案** | 系统推荐哪一项，以及为什么 |
| **用户操作** | 接受推荐 / 选择其他方案 / 自定义补充说明 / 暂不处理 |

### 8.5 决策后的系统行为

一旦用户决策：
1. 写入 decisions.jsonl
2. 相关 workflow 恢复
3. 后续任务读取该结果继续执行

## 9. Reports & Trace 详细设计

### 9.1 为什么要合并成一个模块

报告和追踪本质相关：
- **报告**告诉你"结果怎么样"
- **Trace** 告诉你"它是怎么做出来的"

### 9.2 Reports 子页

建议按章节展示：

| 报告等级 | 内容 |
|---------|------|
| 轻量报告 | 字数、角色数、地点数、beats 推进、风险摘要 |
| 标准报告 | 一致性摘要、节奏摘要、对话比例、结构问题、修改建议 |
| 深度报告 | 风格偏差、角色语气偏差、情绪曲线、章节功能评估 |

**报告必须可配置**。建议等级：Off / Light / Standard / Deep，默认 Light。

### 9.3 Trace 子页

每个任务显示：
- task id、workflow 类型、执行角色链
- 输入文件、输出文件
- 检查结果
- 是否触发决策点
- 最终状态

### 9.4 Trace 的意义

极大增强信任。否则作者会疑惑：
- 你到底读没读 world？
- 你为什么说这里冲突？
- 你凭什么建议改这条伏笔？

## 10. Settings 详细设计

### 10.1 页面目标

把系统可配置的关键行为暴露给作者，而不是藏在文档里。

### 10.2 核心设置项

| 分组 | 设置项 |
|------|--------|
| **Writing** | 每章最小字数、是否必须结尾钩子、对话密度偏好、风格优先级 |
| **Review** | 是否启用一致性检查、是否启用 timeline 检查、报告等级、是否启用深度风格分析 |
| **Autonomy** | L1 / L2 / L3 / L4、哪些情况必须人类决策 |
| **Platform** | OpenClaw / Claude / Codex / API mode |
| **Cost / Model** | 默认模型、高成本任务是否自动运行、优先省钱 / 平衡 / 质量 |

## 11. 哪些是 UI，哪些必须先有底层能力

### 11.1 纯 UI 层可以直接做的

这些主要是展示和交互：
- 页面布局、列表、卡片、tab
- 搜索和筛选
- diff 展示
- 颜色标记、风险图标

### 11.2 必须有底层状态层才能做好的

| UI 功能 | 底层依赖 |
|---------|---------|
| Chapter Studio 的计划页 | Planner 有结构化 plan |
| 风险提示 | Editor 有结构化 issues |
| 本章角色/地点/伏笔 | index |
| Timeline 页 | timeline index |
| Decision Center | decision records |
| Reports | report schema / task records |
| Trace | workflow task logs |

### 11.3 开发顺序建议

**第一步**：先做结构化索引和 workflow 记录

**第二步**：再做最关键的 UI 页：
- Dashboard
- Chapter Studio
- Beat Board

**第三步**：再做：
- Timeline
- Decision Center
- Reports
- Trace

---

## 12. 第一版 UI 的 MVP 建议

**MVP 页面**：Dashboard / Chapter Studio / Beat Board / Settings

### 为什么是这 4 个

| 页面 | 原因 |
|------|------|
| Dashboard | 用户一进来要知道项目状态 |
| Chapter Studio | 这是核心工作页 |
| Beat Board | 这是差异化能力展示页 |
| Settings | 因为强调开箱即用、多平台、AI 友好，配置入口必须可见 |

**第二版再做**：Timeline / Decision Center / Reports / Trace

---

## 13. 一句话收束

> 如果结构化索引是骨架，编排层是神经系统，那 UI 就是：
> **作者和这个小说 Agent 系统之间的驾驶舱。**
>
> 它的意义不是"好看"，而是让作者能：
> **看懂 / 接管 / 修正 / 决策 / 信任**
